Classes vs. Modules

Wed Nov 19, 2014

Classes and modules are very similar. In fact, the class Class is a subclass of the class Module, therefore classes are also module objects. We can think of this as Modules being a more basic structure and classes being a specialization of the this structure.

Both modules and classes are used to encapsulate behaviors and constants. Classes are however are more specialized by also bundling certain behaviour, attributes, and class specific behaviours that will be available only to the class's instances. Modules on the other hand, are bundles of methods that can be accessible accross different classes. Also unlike classes, modules do not have instances.

How do we make methods in classes and modules accessible to objects?

Classes can inherit methods from other classes if that class is it's superclass, or one of their ancestor class(es). Modules on the other hand, get mixed into classes using either the "include" or "prepend" method*. A module mixed into a class this way is often called a "mixin".

*The prepend method is new in Ruby 2.0 and the difference between include and prepend is that when prepend is used the object looks into that module before it does in the class.

So if they are so similar, why bother with whether to decide where to put methods? Why not put them all in modules, or all in classes?

The answer to this question depends on your design. One important point to keep in mind is that classes can only inherit from one superclass. Mixing has a very strong resemblance to inheritance but the main difference is that you can mixin more than one module to a class, as well as mixin the same module to multiple classes.

Therefore, if your design requires numerous behaviours for your class's instances, but you don't want to stash them all in the class, or in the class's superclass and/or ancestor classes, using modules will be a good way to organize your code. Also behaviours that you identify as possible to be exhibited by more than one class or object would be an appropriate candidate for a module.
